## Engine quality gates

Auto-runnable subset (every gate is a `cargo` invocation operators can replay):

```bash
# Functional correctness
cargo test --lib                              # >2700 unit tests
cargo test --test 'e2e_*'                     # >40 integration tests
cargo test --test 'chaos_*'                   # 8 chaos tests
cargo test --test 'drill_*'                   # 3 backup/restore drills

# Build artifacts
cargo build --release                         # Release binary builds
docker build .                                # Multi-stage container builds

# Style + lint
cargo check
cargo clippy --all-targets -- -D warnings
cargo fmt --check
```

Performance gate (post Phase 12, requires `reddb-benchmark` repo):

```bash
cd ~/Work/FF/reddb-benchmark
make duel-official                            # 10 fresh-Docker runs
make check-baseline OFFICIAL_ONLY=1 GATE_THRESHOLD=0.10
make check-baseline OFFICIAL_ONLY=1 GATE_P99_THRESHOLD=0.15
```

## Items deferred past v1.0

Documented in PLAN.md "Out of Scope (Intentional)" — none of these block v1.0:

- Database registry + multi-tenant control plane (separate project)
- Customer auth (OAuth, MFA, sign-up)
- Billing integration
- Dashboard SPA
- DNS routing / wildcard certs
- Cloud-specific provisioning (Fly API, Kubernetes API calls)
- Multi-region replicas with global routing
- Window functions, FDW, full SSI, full extensions
- Automatic failover (manual only in v1.0)
- Pager-level encryption-at-rest (foundation only in v1.0; v1.1 candidate)

## Sign-off

The release manager signs by:

1. Verifying every ✅ row above against the cited evidence on the v1.0 commit.
2. Documenting workarounds for every 🔶 row (or upgrading to ✅ before tagging).
3. Filing follow-up issues for every ❌ row, tagged `v1.x`.
4. Tagging `v1-cloud-agnostic` and bumping `Cargo.toml` version.

## Path to v1.0

The remaining work to flip every 🔶 / ❌ to ✅, in suggested order:

1. **Phase 9.1 cold-start measurement** — run the new harness and publish numbers. May reveal Phase 9.2 (incremental snapshot) is mandatory or merely nice-to-have.
2. **Phase 7.2 size measurements** — let the release artifact-size gate produce the official binary/image rows.
3. **Phase 8.4 matrix expansion** — expand beyond the current Ubuntu + Windows + macOS + MinIO coverage to aarch64 / btrfs / xfs as capacity allows.
4. **G2/G3/G4/G5 hardening** — fuzz parsers, remote-backend conformance, two-region drill, upgrade/downgrade fixtures.
5. **Phase 12 perf program** — gated on the official duel run; only the worst measured deltas get attacked, no speculative work.
