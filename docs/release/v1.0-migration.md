# RedDB v1.0 Migration Guide

This guide is the client-facing checklist for upgrading a v0.x deployment to the v1.0 release line.

## 1. Snapshot Before Upgrade

Before replacing binaries:

```bash
curl -X POST http://<host>:<port>/admin/backup \
  -H "Authorization: Bearer $RED_ADMIN_TOKEN"
red doctor --bind <host>:<port> --json
```

Record:

- Current engine version.
- `wal.current_lsn` from `/admin/status`.
- Last successful backup timestamp.
- Remote backend prefix and bucket/path.

## 2. Environment Variables

Use the public `RED_*` namespace in manifests:

| v1.0 variable | Purpose |
|---------------|---------|
| `RED_BACKEND` | `s3`, `fs`, `http`, or `none`. |
| `RED_REMOTE_KEY` | Remote object key for the database. |
| `RED_S3_ENDPOINT`, `RED_S3_BUCKET`, `RED_S3_REGION`, `RED_S3_PATH_STYLE` | S3-compatible backend config. |
| `RED_S3_ACCESS_KEY_FILE`, `RED_S3_SECRET_KEY_FILE` | Secret-file form for S3 credentials. |
| `RED_HTTP_BACKEND_URL`, `RED_HTTP_BACKEND_PREFIX`, `RED_HTTP_BACKEND_AUTH_HEADER_FILE` | Generic HTTP backend config. |
| `RED_HTTP_CONDITIONAL_WRITES` | Required for HTTP writer leases. |
| `RED_LEASE_REQUIRED`, `RED_LEASE_PREFIX`, `RED_LEASE_TTL_SECS`, `RED_INSTANCE_ID` | Serverless writer lease/fencing. |
| `RED_SLO_REPLICA_LAG_BUDGET_SECONDS` | Replica lag SLO budget, default 60. |

Legacy `REDDB_*` aliases are still accepted where existing code already shipped them, but v1.0 documentation and reference manifests use `RED_*`.

## 3. Writer Lease Requirement

If more than one runtime can attach to the same remote database key, set `RED_LEASE_REQUIRED=true`.

Eligible backends:

- `s3`: recommended for serverless and ephemeral runtimes; uses ETag + conditional requests.
- `fs`: acceptable only on durable local disk / single-writer mounted volumes.
- `http`: acceptable only when the service supports ETag, `If-Match`, and `If-None-Match`, with `RED_HTTP_CONDITIONAL_WRITES=true`.

Ineligible or unsafe backends fail closed during lease acquire.

## 4. Encryption Claim

Do not treat v1.0 as encrypted-at-rest unless the deployment adds external storage encryption (cloud volume encryption, bucket-side encryption, dm-crypt, etc.). RedDB v1.0 includes crypto/page-encryption foundation, but pager-level encrypted file format is not the v1.0 GA promise.

## 5. Upgrade Order With Replicas

1. Upgrade replicas first, one at a time.
2. Wait for `replica.apply_health == "ok"` and replica lag within budget.
3. Promote a known-good replica manually if doing a rolling primary replacement.
4. Restart the old primary as a replica.
5. Run one restore drill from the newest backup before declaring the upgrade complete.

## 6. Rollback/Downgrade

Downgrades are allowed only when the newer binary has not written a newer file-format version. If a future v1.x bumps the format version, older binaries must refuse to open it fail-closed.

For v1.0 rollout, keep the pre-upgrade backup until:

- The v1.0 binary has run a successful backup.
- A restore drill validates the new backup.
- Replica apply health is clean.
